# Intake file schema (`.devblueprint-intake.yml`)

An **intake file** is a small, reproducible record of the answers
`devblueprint init` needs to scaffold a project. Instead of remembering a line of
flags, an agent (or a human) captures the decisions once - from a setup interview
or a PRD - and feeds them to the CLI:

```bash
# Preview exactly what init would write, without touching disk:
devblueprint plan --target ~/Projects/myapp --from .devblueprint-intake.yml

# Apply it once the plan is confirmed:
devblueprint init --target ~/Projects/myapp --from .devblueprint-intake.yml
```

The file is the machine-readable interface later agent flows target: the setup
interview (P3-2) and the PRD importer (P3-3) both end by *writing* one of these,
and `init --from` *consumes* it. It is a plain file the user owns - no lock-in.

A ready-to-copy, annotated template lives at
[`agent/intake.example.yml`](../../agent/intake.example.yml).

## Format

A flat map of `key: value` pairs - deliberately **not** general YAML, so the kit
stays dependency-free (a tiny built-in reader handles it):

- One `key: value` per line.
- Blank lines and comments are ignored. A comment is a line starting with `#`, or
  a trailing ` #...` after a value.
- Values may be single- or double-quoted; quotes are stripped. Quote a value only
  if it has leading/trailing spaces you need to keep.
- Unknown keys are a **hard error** (so typos surface instead of being ignored).
- Every key is optional; an omitted key falls back to init's default.

## Keys

| Key         | Maps to        | Default                    | Notes |
|-------------|----------------|----------------------------|-------|
| `name`      | `--name`       | `--target` basename        | Human-facing project name; fills `{{PROJECT_NAME}}`. |
| `variant`   | `--variant`    | *(none - required)*        | Stack variant slug; must match a `devblueprint list` entry. |
| `main`      | `--main`       | `master`                   | Always-deployable branch. |
| `base`      | `--base`       | `develop`                  | Integration branch. `base == main` selects the single-branch (trunk) flow. |
| `community` | `--community`  | `false`                    | Boolean: `true`/`false` (also `yes`/`no`, `on`/`off`, `1`/`0`). Opts into `SECURITY.md` + `CODE_OF_CONDUCT.md`. |
| `contact`   | `--contact`    | `INSERT CONTACT METHOD`    | Reporting address for the community-health files; only meaningful with `community: true`. |
| `deploy`    | `--deploy`     | *(unset)*                  | Intended deploy target (e.g. `vercel`, `fly`, `docker`, `none`). Recorded and echoed; the runbook is generated by a later wave. |

`--target` is **not** a key: it says *where* to scaffold, not *what*, so it stays
a CLI argument on every invocation.

## Precedence

For each answer the resolution order is:

1. **Explicit CLI flag** (highest) - lets a conversation revise one answer without
   editing the file: `init --from intake.yml --base master`.
2. **Intake file value**.
3. **Built-in default** (lowest).

Because flags win, the same intake file can be reused across runs with a one-off
override, and re-running `plan` shows the net result before anything is written.

## Example

```yaml
name: MyCoolApp
variant: web-nextjs
main: main
base: dev
community: true
contact: security@mycool.dev
deploy: vercel
```

```bash
$ devblueprint plan --target ~/Projects/mycoolapp --from .devblueprint-intake.yml
Plan: init would scaffold 'MyCoolApp' (Next.js web app) into /Users/me/Projects/mycoolapp
  branches: dev -> main   gate: ...
  deploy target: vercel (recorded; runbook lands in a later wave)

  would write docs/engineering/git-workflow.md
  ... (every file init would create) ...

Plan only - nothing was written. Apply it with:
  devblueprint init --target /Users/me/Projects/mycoolapp --from .devblueprint-intake.yml
```

`plan` is exactly `init --dry-run`: it runs init's real code path with every disk
write short-circuited to a `would ...` line, so the preview can never drift from
what `init` actually does.
